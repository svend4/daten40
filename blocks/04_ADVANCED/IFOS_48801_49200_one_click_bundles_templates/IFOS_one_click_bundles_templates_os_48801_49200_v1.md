# IFOS 48801–49200 — One‑Click Bundles & Templates OS (v1)
Цель: превратить “набор модулей” из Registry (48401–48800) в **устанавливаемые и запускаемые пакеты**:
- ZIP / Docker Compose / Helm / Terraform (где уместно)
- импорт/экспорт в Make и n8n
- WordPress bundles (набор плагинов + конфиги)
- preflight‑проверки до установки
- миграции и rollback
- lockfile для повторяемости

Порядок: простое → среднее → сложное.

---

## 48801–48840 — BundleManifest: что такое “бандл” технически
**BundleManifest**:
- id, имя, цель (goal)
- список items (ссылки на registry item ids)
- требования (cpu/ram, домен, порты)
- secrets (что нужно: API keys)
- install targets (docker/wp/make/n8n/k8s)
- тесты (smoke tests)
- exports (zip/compose/helm)

Manifest — “паспорт” бандла.

---

## 48841–48890 — Templates: шаблоны как повторяемые конфигурации
**Template**:
- параметризуемые поля (placeholders)
- presets (варианты: simple/advanced)
- генерация файлов: .env, config.yaml, make blueprint json, n8n workflow json
- документация “как настроить”

Шаблон — это “макрос” на уровне конфигурации.

---

## 48891–48930 — Preflight checks: проверка ДО установки
**PreflightCheck**:
- проверка совместимости версий (WP/PHP/Node/Python)
- проверка доступа к ресурсам (порт свободен, домен настроен)
- проверка секретов (есть ли ключи)
- проверка внешних зависимостей (DB, Redis)
- предупреждения, но без “молчаливых” отказов

Это отвечает на проблему “поставил — не работает”.

---

## 48931–48970 — InstallPlan: план установки как исполняемый граф
**InstallPlan**:
- шаги (download, verify, configure, deploy)
- порядок (dependencies first)
- параллельность (что можно ставить параллельно)
- точки восстановления (checkpoints)
- пост‑настройка (create admin, create webhook)

План нужен, чтобы “одна кнопка” была надежной, а не магией.

---

## 48971–49010 — Lockfile: повторяемость и “не сломалось через неделю”
**BundleLockfile**:
- зафиксированные версии всех items
- хэши артефактов
- источники загрузки
- совместимость, которая была проверена
Если lockfile есть — можно повторить установку 1:1.

---

## 49011–49050 — SecretsPlan: управление ключами безопасно
**SecretsPlan**:
- где хранить (vault / env / secret manager)
- кто имеет доступ (roles)
- ротация ключей
- маскирование в логах
- запрет попадания в экспортные zip

Секреты — главный источник утечек в “народных сборках”.

---

## 49051–49090 — MigrationPlan: обновления и миграции данных
**MigrationPlan**:
- версия → версия (semver)
- миграции БД/конфигов
- backward‑compat notes
- dry‑run (проверка без применения)

Бандлы живут долго: без миграций всё развалится.

---

## 49091–49130 — RollbackPlan: как откатить без боли
**RollbackPlan**:
- что откатываем (контейнеры/плагины/конфиги/БД)
- точки восстановления (backup snapshot)
- rollback steps
- когда rollback запрещён (если нарушит целостность)
Откат — часть “инженерной культуры”, а не аварийная импровизация.

---

## 49131–49170 — ExportJob: экспорт под разные среды
**ExportJob**:
- формат: zip/compose/helm
- включать ли demo data
- включать ли docs/examples
- генерация “receipt” (что установлено)
Экспорт — ключ к “компоновке и распространению” рационализированных решений.

---

## 49171–49200 — Install Receipt + Compatibility Report: отчёт пользователю
**InstallReceipt**:
- что поставлено, версии, где запущено
- ссылки на UI
- health status
**CompatibilityReport**:
- что проверено, что рискованно, что не соответствует

Это делает “интернет‑макросы” профессиональным продуктом.

---

## Итог
Бандлы = ваша идея “не изобретать новое, а собирать” → в инженерном виде:
- registry (словарь) + one‑click (установка) + preflight + lockfile + rollback
