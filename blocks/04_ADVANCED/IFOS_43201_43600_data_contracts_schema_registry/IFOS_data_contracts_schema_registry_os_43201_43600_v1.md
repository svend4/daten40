# IFOS 43201–43600 — Data Contracts & Schema Registry OS (v1)
Цель: превратить “дикий интернет” в **управляемый рынок данных и интеграций**.
Чтобы миллионы приложений и коннекторов могли **надежно стыковаться**, нужны:
- единые **контракты данных** (data contracts)
- **реестр схем** (schema registry) с версиями
- правила **совместимости** (compatibility)
- **миграции** и преобразования
- валидация (ingest/egress) и **runtime enforcement**
- тесты контрактов (contract tests)

Порядок: простое → среднее → сложное.

---

## 43201–43230 — Что такое Data Contract простыми словами
Контракт данных — это “обещание” между:
- источником (интеграцией/партнёром/сервисом)
- и потребителем (workflow/агент/витрина/другое приложение)

Контракт фиксирует:
- какие поля есть
- какие обязательны
- какие типы и форматы
- какие значения допустимы
- какие версии поддерживаются
- какие гарантии качества (например: уникальность, полнота)

---

## 43231–43280 — Schema Record: единица реестра
**SchemaRecord** описывает одну схему:
- schema_id (например: `ifos.invoice`)
- version (SemVer: 1.2.0)
- формат (JSON Schema / Avro / Protobuf)
- owner/team
- статус (draft, active, deprecated)
- ссылки на примеры и тесты

Это “паспорт” интерфейса данных.

---

## 43281–43330 — Schema Registry: реестр + индекс
**SchemaRegistry** хранит:
- список SchemaRecord
- поиск по namespace/category
- политики версий (например: держим 3 активные версии)
- “канонические” примеры

Без реестра интеграции ломаются при каждом обновлении.

---

## 43331–43390 — Compatibility Rules: правила совместимости
Нужно различать:
- backward compatible (новый продюсер → старый консьюмер)
- forward compatible (старый продюсер → новый консьюмер)
- full compatible (оба направления)

Типовые правила:
- добавлять поля можно, если они optional или имеют default
- удалять поля — major version
- менять типы — major version
- расширять enum — обычно minor, но зависит от потребителя

Выход: **CompatReport** (что можно, что нельзя, почему).

---

## 43391–43440 — Migration Plan: план миграции и трансформации
Если нужно нарушить совместимость — делаем **MigrationPlan**:
- какие версии затронуты
- какие поля меняются
- преобразование (mapping) old → new
- дедлайн отключения старой версии
- rollout strategy (dual-write, shadow-read)

---

## 43441–43490 — Validation Result: валидация данных при входе/выходе
Валидация нужна в двух местах:
1) ingest (когда данные входят в IFOS)
2) egress (когда IFOS отдаёт наружу)

**ValidationResult**:
- ok / errors / warnings
- список нарушений (path, expected, got)
- severity + suggested fix

Это повышает качество данных и уменьшает хаос.

---

## 43491–43530 — Runtime Enforcement Policy: принуждение на рантайме
Не всё можно “поймать” на входе. Нужны политики:
- reject (отклонить)
- quarantine (в карантин)
- coerce (попытаться привести тип: “123” → 123)
- redact (замаскировать PII)
- fallback (подставить default)

**EnforcementPolicy** привязывается к интеграции/категории/тенанту.

---

## 43531–43570 — Contract Tests: тесты, которые защищают рынок
**ContractTest**:
- schema_id + version range
- payload samples
- ожидаемые результаты (pass/fail)
**ContractSuite** объединяет тесты для релиза интеграции.

Это делает “сертификацию” технически проверяемой.

---

## 43571–43600 — Итог и как использовать
Этот блок превращает IFOS в “операционную систему данных”:
- схемы и версии видны и контролируются
- совместимость проверяется до релиза
- миграции планируются, а не случаются
- данные валидируются и приводятся к порядку
- контрактные тесты защищают экосистему
