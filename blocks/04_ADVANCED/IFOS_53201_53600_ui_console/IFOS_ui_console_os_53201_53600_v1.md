# IFOS 53201–53600 — UI Console OS (Admin Console: “почему не работает” + управление) (v1)
Цель: сделать “панель управления интернет‑функциями”, как:
- Диспетчер задач (что запущено)
- Системный журнал (что сломалось)
- Центр обновлений (что установить)
- Центр согласований (что утвердить)
- Центр затрат (сколько стоит)
- Центр данных (что хранится)
- Центр безопасности (кто что сделал)

Порядок: простое → среднее → сложное.

---

## 53201–53230 — Информационная архитектура консоли (как в Office/ERP)
Основные разделы:
1) Home (обзор)
2) Catalog (bundles/templates)
3) Runs (запуски/джобы/процессы)
4) Approvals (согласования)
5) Conflicts (конфликты данных/прав/версий)
6) Data (сущности, словари, lineage)
7) Costs (бюджеты/лимиты)
8) Logs & Traces
9) Alerts & Incidents
10) Security & Audit
Важное правило: **одно место = одна правда** (no “дублирующих вкладок”).

---

## 53231–53270 — Status Model: единая модель статусов
Статусы объектов:
- bundle: installed/needs_update/broken
- process: active/paused/failed
- connector: ok/auth_required/rate_limited
- run: running/succeeded/failed/needs_human
- data: healthy/drift/schema_mismatch
- policy: enforced/warn_only/blocked
Каждый статус должен иметь:
- “что это значит”
- “что сделать”
- “кто отвечает”
Это превращает ошибки в задачи.

---

## 53271–53310 — Dashboards: минимальный набор панелей
Home dashboard:
- “сегодня” (runs, failures, approvals)
- “стоимость” (месяц/день)
- “качество данных” (dedup, errors)
- “безопасность” (policy blocks)
- “интеграции” (auth_required, 429)
Панели должны быть “кликабельными” до причины (drill‑down).

---

## 53311–53350 — Runs/Jobs View: лента запусков как логика производства
Для каждого запуска:
- вход (trigger)
- шаги процесса (timeline)
- какой bundle/версия
- какие данные затронул
- сколько стоило
- где остановился и почему
Кнопки:
- retry
- resume
- run with debug
- export run report (для поддержки/аудита)

---

## 53351–53390 — Approvals Center: “входящие” согласования
UI как почта:
- список approvals
- фильтры (по роли, дедлайну, типу)
- карточка: контекст + diff + риск + политика
Действия:
- approve / reject
- request changes (комментарий)
- delegate (передать другому)
- escalate (если дедлайн)

Важно: approvals должны объяснять последствия:
“Если отклонить — публикация не произойдёт”.

---

## 53391–53430 — Conflicts & Resolution: конфликты должны быть видимы
Типы конфликтов:
- schema conflict (поле поменялось)
- identity conflict (дубликаты сущностей)
- permission conflict (нет прав)
- version conflict (bundle несовместим)
UI показывает:
- что конфликтует
- чем отличается (diff)
- предлагаемые решения (merge, choose A, choose B)
- безопасный режим (freeze до решения)

---

## 53431–53470 — Logs/Traces: “человеческий” режим
Два режима:
- Simple (для пользователя): “что произошло”
- Expert (для инженера): request/response, trace_id, payload hints
Функции:
- поиск по trace_id/run_id
- фильтр по connector/bundle
- “скопировать отчёт для поддержки” (без секретов)
Важно: скрывать PII и секреты по умолчанию.

---

## 53471–53510 — Cost & Budgets: контроль денег и лимитов
UI показывает:
- cost per run / per bundle
- budget per month
- лимиты токенов LLM
- топ‑дорогие процессы
Действия:
- поставить cap
- включить degrade mode (без LLM)
- расписание “не запускать ночью”
Это делает систему управляемой для малого бизнеса.

---

## 53511–53540 — Catalog/Marketplace внутри консоли
В консоли можно:
- искать bundles/templates
- смотреть рейтинги и совместимость
- устанавливать “one click”
- видеть зависимости (как npm)
- видеть политики (что требуется)
- rollback/upgrade
Это “App Store” для функций.

---

## 53541–53570 — Security & Audit: кто и что сделал
Раздел:
- audit events (logins, approvals, installs)
- policy blocks (что было заблокировано)
- secret rotations
Плюс: экспорт для комплаенса (PDF/CSV).

---

## 53571–53590 — Alerts & Incidents: реагирование
Alert:
- что случилось
- severity
- impacted processes
- suggested action
Incident:
- timeline
- ownership
- runbook links
- postmortem template
Цель: любой сбой превращается в управляемый процесс.

---

## 53591–53600 — Accessibility (A11y) и “office‑style” UX
Требования:
- клавиатурная навигация
- контраст
- крупные кликабельные зоны
- “объяснить простыми словами”
Три уровня текста:
- коротко
- подробно
- техподробности
Это важно, чтобы консоль была “для людей”, а не только для инженеров.
