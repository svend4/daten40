# IFOS 59601–60000 — Enterprise Rollouts & Change Control OS (внедрение и управление изменениями) (v1)
Цель: чтобы IFOS был не “песочницей”, а **B2B‑операционной системой**: обновления не ломают бизнес, всё проверяемо и откатываемо.
Это то, чего обычно не хватает “плагинам и сценариям”: они работают у автора, а у компании — ломаются из‑за процессов.

Порядок: простое → среднее → сложное.

---

## 59601–59630 — Базовая модель сред (dev → staging → prod)
Минимальная дисциплина:
- dev: эксперименты
- staging: близко к прод, тестовые данные
- prod: реальные данные и SLA
Переходы между средами — только через “promotion” (см. ниже).

---

## 59631–59670 — Release trains (релизные поезда)
Release train = расписание релизов:
- weekly train (каждую неделю)
- monthly train (для критичных систем)
- emergency hotfix lane (вне расписания)
Зачем: пользователи и IT заранее знают, когда ждать изменений. Это снимает страх обновлений.

---

## 59671–59710 — Approval gates (ворота одобрения)
В B2B нельзя “залить в прод одним кликом” без контроля.
Gates:
1) docs ready (паспорт + runbook)
2) benchmark passed
3) security scan passed
4) conformance suite passed
5) business owner approval (не технарь, но владелец процесса)
6) change window (время, когда можно менять)
Без gate — нет продвижения в prod.

---

## 59711–59750 — Environment promotion (продвижение по средам)
Promotion = артефакт с фиксированными версиями:
- bundle version
- connector versions
- runtime version
- configs (as code)
Promotion создаёт “точку восстановления”: можно откатиться к известному состоянию.

---

## 59751–59790 — Migration playbooks (миграции данных/схем)
Когда меняется схема:
- migration plan (шаги)
- backward compatibility (поддержка старого)
- dual write / dual read (переходный период)
- data backfill (перезаполнение)
- verification queries (проверки)
- cutover + rollback point
Это особенно важно для “порталов сравнения” и “отзывов”, где данные живут годами.

---

## 59791–59820 — Rollback strategies (откат)
Откат обязателен:
- instant rollback (переключить версию)
- blue/green (две параллельные версии)
- canary (1–5% пользователей)
- feature flags (см. ниже)
Rollback должен быть не “ручной магией”, а кнопкой с runbook‑описанием.

---

## 59821–59850 — Feature flags (фичефлаги)
Новые функции включаются:
- по процентам (canary)
- по группам (pilot users)
- по регионам/языкам
- по типу клиента (SMB vs enterprise)
Так вы можете внедрять изменения без риска “сломать всех”.

---

## 59851–59880 — Change management (управление изменениями)
Документ “Change Request”:
- что меняем
- почему
- риски
- план
- окно изменений
- ответственные
- план отката
Это бюрократия, но “правильная”: она защищает бизнес и репутацию.

---

## 59881–59910 — Audit & Controls (аудит и контроль)
Для компаний важно:
- кто изменил конфиг
- кто дал approval
- когда произошёл rollout
- что именно обновилось
- какие данные затронуты
Audit лог + подписи (tamper‑evident) дают доверие и соответствие compliance.

---

## 59911–59940 — SLO/SLA (обязательства качества)
Определяем:
- SLO: цели (например, 99.5% успешных запусков)
- SLA: контрактные гарантии
- error budget: сколько можно “плохих” запусков
Если бюджет исчерпан — релизы замораживаются, пока не восстановится качество.

---

## 59941–59970 — Incident response (инциденты)
Если что-то пошло не так:
- автоматический алерт
- triage (класс инцидента)
- rollback (если нужно)
- postmortem (почему и как избежать)
- обновление runbook
Инцидент = источник улучшения системы, а не “стыд”.

---

## 59971–60000 — Enterprise templates (шаблоны внедрения)
Готовые шаблоны:
- Rollout plan (канарейка → 100%)
- Migration plan (schema v1→v2)
- Change request
- Postmortem
- Monthly release report
Это делает IFOS пригодным для реальных организаций (палаты, ведомства, компании).

---

## Итог
Change Control OS превращает IFOS в “операционную систему бизнеса”:
- безопасные обновления
- проверки и approvals
- миграции и откаты
- аудит и SLA
Это закрывает разрыв между “есть тысяча плагинов” и “можно внедрить и жить”.
