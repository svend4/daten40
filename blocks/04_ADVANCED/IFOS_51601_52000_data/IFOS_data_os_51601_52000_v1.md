# IFOS 51601–52000 — Data OS (Entities, MDM, Sync, Versioning, Schema Evolution) (v1)
Цель: остановить “дикое поле данных” (миллионы CSV/таблиц/плагинов, где всё разное).
Data OS делает:
- единые сущности (entities) и типы
- канонические записи (source of truth)
- дедупликацию и “склейку” идентичностей
- синхронизацию с конфликт‑резолвом
- версионность и lineage (происхождение)
- эволюцию схем (contracts)
- качество и правила
Порядок: простое → среднее → сложное.

---

## 51601–51640 — Минимальная модель: Entity + Record
**Entity** = “реальная штука” (человек, компания, документ, продукт, интеграция).
**Record** = “как эта штука выглядит в конкретной системе” (CRM, WP, Make, Excel).

Простейший слой:
- entity_id (стабильный)
- record_id (в системе‑источнике)
- source_system
- payload (данные)
- timestamps
Так мы можем собирать всё из разных мест и не путать.

---

## 51641–51690 — Entity Types: словарь сущностей (как “таблица Менделеева” для данных)
Нужен реестр типов:
- Person, Organization, Project
- Document, File, Message
- Product, Service, Offer
- Connector, Bundle, TemplatePack
- Event (событие), Ticket (тикет), Review (отзыв)
Каждый тип имеет:
- обязательные поля
- правила качества
- маппинги из популярных источников

---

## 51691–51730 — MDM: Source of Truth и каноническая запись
**MDM (Master Data Management)**:
- выбираем “главный источник” для сущности (или правила выбора)
- формируем canonical_record
- сохраняем ссылки на исходные records
Плюс:
- “переопределение” (manual override) для важных полей
- “доверие к источникам” (trust weights)

---

## 51731–51780 — Dedup & Identity Linking: склейка одинаковых сущностей
Проблема: один человек = 5 записей (email/CRM/телефон).
Решение:
- ключи сопоставления (email, phone, domain, external_id)
- fuzzy matching (имя/адрес)
- merge decisions (авто или ручной)
Результат: у сущности один entity_id и “граф ссылок” на всё.

---

## 51781–51820 — Sync & Conflict Resolution: как не ломать данные
Синхронизация:
- pull (забрать)
- push (отдать)
- bidirectional (двунаправленно)
Конфликты:
- last write wins (опасно)
- source priority (источник важнее)
- field-level merge (разные поля из разных источников)
- manual resolve (критические поля)
В IFOS: конфликт — отдельный объект, а не “тихо перетёрли”.

---

## 51821–51860 — Versioning & Lineage: откуда что взялось
Версионность нужна для:
- аудита и доверия
- rollback
Lineage:
- какой коннектор/бандл создал запись
- какой трансформ применили
- какие источники использовались
Это “протокол происхождения данных”.

---

## 51861–51910 — Schema Contracts & Evolution: схемы должны взрослеть
Schema contracts:
- versioned schema (v1, v1.1, v2)
- backward compatibility правила
- migration scripts
- validation on ingest
Так исчезает “сломали API — всё упало”.

---

## 51911–51950 — Data Quality Rules: качества без “ручной уборки”
Правила:
- required fields
- ranges (например цена >= 0)
- format checks (email, url)
- referential integrity (ссылка на существующую сущность)
- anomaly detection (внезапный скачок значений)
В IFOS качество — вычисляемый сигнал, влияющий на рейтинг модулей и витрин.

---

## 51951–51980 — Event Sourcing / Outbox: надежная доставка изменений
Для устойчивой синхронизации:
- outbox pattern (изменения сначала в журнал)
- идемпотентность (повтор не ломает)
- event_id и dedup keys
Это критично при большом числе коннекторов.

---

## 51981–52000 — Exports & Snapshots: “снимок истины” для обмена
Экспорт:
- snapshot пакета сущностей
- manifest + schema versions
- hash/signature (для доверия)
- диффы (что изменилось)
Это делает возможным “перенос организации” или “переезд на другой хостинг”.
