# IFOS 48401–48800 — App/Plugin/Connector Registry OS (public catalog) (v1)
Цель: “каталог всего интернета” в практическом виде: **плагины / приложения / коннекторы / модули / интеграции**.
Ваш тезис: “есть миллионы примеров кода и приложений, но они не интегрированы и не понятны”.
Регистри — это слой, который превращает хаос в “словарь” + “витрины” + “одна кнопка”.

Порядок: простое → среднее → сложное.

---

## 48401–48440 — RegistryItem: единая карточка модуля
**RegistryItem** (что именно записываем):
- id, имя, тип (app/plugin/connector/workflow/template)
- краткое описание (1–3 строки)
- capability tags (что умеет)
- входы/выходы (I/O)
- ссылки: репозиторий, сайт, документация
- лицензия и условия использования
- поддержка/maintenance (частота релизов, issue tracker)

Карточка должна быть **стабильной** и пригодной для поиска/сравнения.

---

## 48441–48490 — Capability Vocabulary: “словарь функций”
Чтобы строить кластеры, нужен словарь:
- auth.oauth2, auth.api_key
- data.fetch.rss, data.fetch.http, data.parse.html
- docs.pdf.extract, docs.docx.parse
- automation.webhook, automation.scheduler
- ai.rag.retrieve, ai.llm.call
- storage.s3, storage.gdrive

Это позволяет: “покажи всё, что делает X” и “подбери аналоги”.

---

## 48491–48530 — Compatibility Matrix: совместимость и версии
**CompatibilityMatrix**:
- платформа: WordPress 6.x, Node 20, Python 3.11, Make, n8n, Zapier
- окружение: self‑hosted / cloud
- OS/архитектура: Linux/Windows/ARM
- “известные конфликты” (plugin A ломает plugin B)

Без матрицы совместимости “одна кнопка” невозможна.

---

## 48531–48570 — Dependencies: зависимости и “пакеты решений”
**Dependency**:
- требует ли модуль базу данных?
- требует ли ключи API?
- зависит ли от другого плагина/коннектора?
- есть ли миграции и как откатить?

На базе зависимостей формируются bundles (см. 48571–48610).

---

## 48571–48610 — InstallProfile / RunProfile: “одна кнопка”
**InstallProfile**:
- шаги установки (docker compose / npm / wp install)
- переменные окружения
- необходимые ключи
**RunProfile**:
- как запустить
- healthcheck
- тестовый запрос
- пример конфигурации

Это превращает “README из GitHub” в машинно‑исполняемую инструкцию.

---

## 48611–48650 — Security Profile: доверие и риски
**RegistrySecurityProfile**:
- уровни доверия: community / verified / audited
- SCA‑скан, подписи релизов
- какие разрешения требуются (WP capabilities, scopes OAuth)
- обработка секретов
- политика CVE/обновлений

Без этого регистри станет “распространителем троянов”.

---

## 48651–48690 — DocsPack + ExamplesPack: документация как продукт
**DocsPack**:
- краткая справка
- детальный гайд
- типовые ошибки
**ExamplesPack**:
- 3–10 готовых рецептов
- “минимальный пример”
- “боевой пример”

То, чего не хватает WordPress‑плагинам и Make‑шаблонам — системная документация и примеры.

---

## 48691–48730 — Quality Score: качество и “почему это в топе”
**RegistryQualityScore**:
- актуальность (freshness)
- покрытие документацией (docs coverage)
- тестируемость (есть ли e2e)
- security signals
- отзывы (см. 48001–48400)
- стабильность API (semver)

Итог: объяснимое ранжирование “почему это выше”.

---

## 48731–48770 — Tagging & Clustering: кластеры решений
**RegistryTagging**:
- теги по вертикалям (travel, insurance, automation)
- теги по ролям (ingest, transform, store, notify)
- теги по уровню сложности (simple/advanced)
- “сцепки” (часто используется вместе)

Результат: “пакеты решений” (bundle) — то, что вы описывали как рационализацию и компоновку.

---

## 48771–48800 — Bundles: готовые наборы (конфигурации)
**RegistryBundle**:
- состав (items)
- install/run profiles на уровне пакета
- целевая задача (“Сбор новостей + дедуп + конференция”)
- экспорт: zip/compose/terraform
Бандлы — это “макросы интернета”.

---

## Итог
Регистри отвечает на вопрос “почему хаос”:
- нет стандартной карточки
- нет словаря функций
- нет совместимости и профилей установки
- нет качества и прозрачного топа
IFOS делает это стандартом.
