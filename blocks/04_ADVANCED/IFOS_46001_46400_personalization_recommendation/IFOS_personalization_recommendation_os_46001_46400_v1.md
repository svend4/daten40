# IFOS 46001–46400 — Personalization & Recommendation OS (v1)
Цель: сделать “лучшее для меня” из хаоса миллионов функций:
- персональные витрины, подборки и “наборы кнопок”
- рекомендации в контексте задачи (what you are doing now)
- прозрачные объяснения “почему это предложено”
- приватность по умолчанию (privacy-first)
- обучение на фидбеке без превращения в “слежку”

Порядок: простое → среднее → сложное.

---

## 46001–46040 — Профиль: минимальная персонализация без риска
**UserProfile** (минимум):
- язык/регион/часовой пояс
- тип пользователя (solo / team / enterprise)
- любимые категории
- запреты (“не показывать крипто”, “без рекламы”, “только open-source”)

Это даёт пользу без сложного ML.

---

## 46041–46090 — Контекст: “что пользователь делает сейчас”
**ContextSignal**:
- текущая задача (например: “сравнить страховки”)
- платформа (web/mobile)
- ограничения (бюджет, сроки, приватность)
- уже выбранные инструменты (Make/n8n/WordPress)

Контекст — ключ к рекомендациям “по делу”, а не “вообще”.

---

## 46091–46130 — Предпочтения: явные и неявные
**Preference**:
- explicit: пользователь поставил галочки/фильтры
- implicit: клики, установки, “оставил себе”, “скрыл”

Важно: не хранить лишнее, аггрегировать, давать управление пользователю.

---

## 46131–46170 — Recommendation Request/Response: API рекомендации
Рекомендательный сервис должен уметь:
- получить request (profile + context + intent)
- вернуть список items + score + explanation + alternatives
- вернуть “bundles”: готовые наборы интеграций

---

## 46171–46210 — Ranking Features: на чём ранжируем
**RankingFeature** (простые признаки):
- качество (trust tier, badges) (из 45601–46000)
- совместимость (есть нужные коннекторы/платформа)
- скорость внедрения (setup time)
- популярность в похожем сегменте
- стоимость/квоты
- “свежесть” (актуальная версия)
- риск (approval gates, PII, деньги)

Это можно сделать без нейросетей — логикой + weighted score.

---

## 46211–46250 — Explanations: “почему вы это видите”
**RecommendationExplanation**:
- “потому что вы выбрали категорию X”
- “подходит для Make + Google Sheets”
- “Tier2 доверия и Verified Run”
- “низкая стоимость и быстрый setup”

Объяснения резко повышают доверие и снижают раздражение.

---

## 46251–46290 — Cold Start: когда данных нет
**ColdStartPolicy**:
- спрашиваем 3–5 вопросов (персона, цель, платформа)
- предлагаем “Starter Kits” по категориям
- используем редакторские витрины (curation) как базу
- постепенно учимся по явному фидбеку

Cold start решается UX‑ом, а не “супер‑ИИ”.

---

## 46291–46330 — Privacy & Consent: приватность как настройка ядра
**PersonalizationPrivacyPolicy**:
- какие сигналы собираем
- как долго храним
- где храним (on-device / tenant / cloud)
- opt-out по умолчанию для чувствительных сигналов

**ConsentRecord**:
- согласие на персонализацию
- согласие на телеметрию
- согласие на сегментацию

Это критично для EU/GDPR‑логики.

---

## 46331–46370 — Feedback Loop: обучение без “наблюдения”
**FeedbackEvent**:
- like/dislike
- “скрыл”
- “установил/удалил”
- “сработало/не сработало”
- “слишком сложно/дорого”

Используем:
- агрегирование
- дифференциальную приватность (при необходимости)
- обучение на уровне сегментов, не личности

---

## 46371–46400 — A/B тесты и fairness
**ABTest**:
- сравнение стратегий ранжирования
- проверка “не сломали ли рекомендацию”

**FairnessAudit**:
- нет ли дискриминации сегментов
- не душим ли open‑source
- не продвигаем ли платные “в ущерб качеству”

---

## Итог
Персонализация — это не “магическая нейросеть”, а:
- профили + контекст
- ранжирование по качеству/совместимости/стоимости
- объяснения
- privacy-first и consent
- обучение на фидбеке
