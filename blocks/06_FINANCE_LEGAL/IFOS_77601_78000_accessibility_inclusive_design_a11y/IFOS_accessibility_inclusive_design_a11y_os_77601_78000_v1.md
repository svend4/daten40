# IFOS — Доступность и инклюзивный дизайн (A11y) — блок 77601–78000

## 1) Зачем нужен блок
Этот блок вводит **системный слой доступности (Accessibility)** для IFOS: правила, компоненты и проверки, которые гарантируют,
что интерфейсы (консоль, офис‑UI, маркетплейс, мастера настройки, витрины, отчёты, формы) доступны для пользователей с разными
возможностями и условиями (мобильный, слабый интернет, крупный шрифт, клавиатура‑только, скрин‑ридер, повышенный контраст и т. п.).

Фокус: **WCAG‑совместимый дизайн**, практическая проверка (авто+ручная), и «путь исправления» (remediation) как часть релизного цикла.

## 2) Что блок покрывает
### 2.1 Системные настройки доступности (User A11y Profile)
- Профиль: размер шрифта, контраст/тема, плотность интерфейса, уменьшение анимаций, режим «клавиатура‑только».
- Пер‑приложение и глобально (workspace‑политики могут задавать минимальные требования).
- Доступность в мастерах (guided setup), help‑центре, карточках, таблицах, графах, редакторах.

### 2.2 Компонентная библиотека и паттерны
- Базовые компоненты с семантикой: кнопки, меню, вкладки, модальные окна, таблицы/гриды, формы.
- ARIA‑паттерны (role, aria‑label, aria‑describedby, aria‑live), управление фокусом, trap‑focus.
- Клавиатурная навигация: таб‑порядок, shortcuts (с учётом пользовательских конфликтов), skip‑links.
- Контраст, состояния ошибок, видимые фокусы, масштабирование/рефлоу (zoom/reflow).

### 2.3 Контент‑доступность
- Текст альтернативы (alt), подписи/субтитры, транскрипты, понятные заголовки и структуры.
- Проверки для импортируемого контента (документы/вложения) и витрин знаний.
- Политики для PDF/док‑экспорта (теги, структура, язык документа).

### 2.4 Тестирование и «гейт» качества
- Автотесты: lint rules, axe‑core/подобные проверки, unit + E2E (Playwright), Lighthouse‑аудит.
- Ручные проверки: скрин‑ридеры, клавиатура‑только, контраст, zoom 200%+, high contrast.
- Политика «не выпускать» при критических A11y‑дефектах (интеграция с feature flags и progressive delivery).

### 2.5 Аудит, трекинг и исправление
- Единая модель **AuditRun → Issues → RemediationPlan → Fix PR/Release**.
- Приоритизация: критичность/влияние/частота/поверхность (консоль, маркетплейс, витрины).
- Отчёты по командам и по компонентам (в связке с service catalog/topology/ownership).

## 3) Что блок НЕ делает (non‑goals)
- Не заменяет полноценную юридическую экспертизу по отраслевым требованиям (это отдельные блоки).
- Не строит сам скрин‑ридер и не реализует OS‑уровневые функции доступности устройства.
- Не является «единственным» QA: это слой доступности в составе общей системы качества.

## 4) Основные сущности данных
- **A11yProfile**: user_id, workspace_id, preferences (font_scale, contrast_mode, reduce_motion, keyboard_only…).
- **AuditRun**: target (app/page/component), tool (auto/manual), timestamp, summary.
- **A11yIssue**: wcag_ref, severity, repro_steps, impacted_users, component, screenshots/notes.
- **RemediationPlan**: owner_team, milestones, expected_fix, verification_steps, deadline.
- **A11yReport**: агрегаты по командам/релизам/компонентам, статус соответствия.

## 5) Интерфейс API (логика)
- /a11y/profiles (CRUD)
- /a11y/audits (создание и чтение прогонов)
- /a11y/issues (регистрация, статус, привязка к тикетам и релизам)
- /a11y/reports (сводки, экспорт)
- /a11y/policies (workspace‑требования, минимальные настройки, исключения)

## 6) UX‑принципы для IFOS
- «Невидимый» A11y‑каркас: доступность включена по умолчанию, а не «по кнопке».
- Ошибки формы понятны: кратко, конкретно, с подсказкой и доступным фокусом.
- Визуальная ясность: контраст, не только цвет (иконки/подписи), стабильный layout.
- Отдельный режим обучения (help): «покажи как пользоваться клавиатурой/скрин‑ридером».

## 7) Интеграции с другими блоками
- UI Console / Office UI: единые компоненты и shortcuts.
- Guided Setup / In‑App Guidance: доступные подсказки и шаги.
- Feature Flags & Progressive Delivery: блокировка релиза при критических A11y‑ошибках.
- Analytics & Synthetic Monitoring (опционально): A11y‑метрики и «путешествия пользователя».

## 8) Мини‑чеклист внедрения (MVP)
1) Добавить A11yProfile и глобальные настройки UI.
2) Включить базовые lint‑правила и auto‑audit для ключевых страниц.
3) Ввести «Issue→Remediation» поток (интеграция с тикетингом).
4) Настроить CI‑гейт и отчёт по релизу.
5) Провести ручной аудит ключевых сценариев (логин, поиск, покупка/оплата, публикация).
