# IFOS — App Shell & Navigation (Блок 38801–39200)

Версия: v1 · Пакет: `IFOS_38801_39200_app_shell_navigation_os_pack.zip` · Дата: 2026-01-02

## 0) Зачем нужен этот блок

Этот блок делает IFOS **похожим на «офисный пакет»**, где пользователь не тонет в тысячах функций:
- есть **единая оболочка приложения** (App Shell)
- есть **навигация и маршрутизация** (router)
- есть **переключение контекста** (workspace switcher)
- есть **единая модель страниц/видов/панелей**, куда подключаются «витрины», «макросы», «коннекторы», «знания».

UI Builder (38401–38800) умеет **собирать интерфейсы**, но без App Shell всё превращается в набор отдельных экранов.
А Federation (39201–39600) требует, чтобы у клиента уже был понятный механизм **tenant/workspace context**.

## 1) Что появится после внедрения

1) Единый каркас приложения: верхняя панель, боковое меню, контентная область.
2) Маршрутизация: URL → view → permissions → data loading.
3) Переключатель workspace + сохранение последнего контекста.
4) «Командная палитра» (Ctrl+K) и быстрые действия.
5) Единая система уведомлений (toast/alerts) и ошибок.
6) Плагинообразное подключение разделов: витрины, маркетплейс, макросы, знания.

## 2) Границы блока (что НЕ входит)

- Конкретные страницы витрин/маркетплейса/знаний — их делают соответствующие блоки.
- Глубокая аналитика UX/AB тесты — это отдельный слой (эксперименты/аналитика).
- Полная «Office Suite UI» (60801–61200) — это более широкая унификация, здесь — базовый каркас.

## 3) MVP (простое → среднее)

### Шаг 3.1 — Модель View/Route

Минимальная сущность: `view` (экран/страница).
- `view_id`
- `path` (например `/workspaces/:id/vitrines`)
- `title`
- `nav_group` (в какой группе меню)
- `required_permissions` (массив)
- `data_sources` (какие API дергать)

MVP: 10–20 базовых view, чтобы система стала «понятной».

### Шаг 3.2 — Каркас интерфейса (layout)

- TopBar: логотип, поиск, workspace switcher, профиль.
- SideNav: группы меню + закреплённые действия.
- Content: рендер выбранного view.
- Footer (опционально): статус, версия, ссылки.

### Шаг 3.3 — Контекст (Workspace Context)

Ключевое правило: **все запросы** к данным и действия пользователя должны идти в контексте `workspace_id`.
В UI это выглядит как:
- при входе → выбирается personal workspace
- пользователь переключается на team workspace
- URL и состояние приложения запоминают `workspace_id`

### Шаг 3.4 — Проверка прав перед показом view

Алгоритм:
1) router распознаёт view
2) запрашивает `GET /me` + `GET /workspaces` (если ещё нет)
3) вычисляет роль пользователя в workspace
4) сравнивает с `required_permissions` view
5) либо открывает view, либо показывает 403/upgrade screen

## 4) Средний уровень: Command Palette, быстрые макросы, deep links

### Шаг 4.1 — Командная палитра (Ctrl+K)

Цель: пользователь не ищет кнопку среди 1000 функций — он вводит команду.
Команды бывают:
- навигационные (перейти на view)
- действия (создать витрину, запустить макрос)
- справка (открыть документацию блока)

### Шаг 4.2 — Deep links и шаринг

Каждый объект (вitrine, macro, connector, knowledge pack) имеет ссылку:
`/w/{workspace_slug}/<module>/<id>`
Это резко повышает «склеенность» системы.

### Шаг 4.3 — Единая система уведомлений

Классы уведомлений:
- success/info/warn/error
- long-running tasks (progress)
- security warnings (подозрительные входы, токены)

## 5) Ближе к сложному: мульти-тенант и federation readiness

Подготовка к 39201–39600:
- App Shell должен поддерживать несколько «тенантов» / окружений (например EU/US/Local)
- переключение источников данных (base URL) через profile/env
- отображение «где мы сейчас» (tenant badge)

## 6) API/контракты (минимально)

- `GET /me` (пользователь + settings)
- `GET /workspaces` (список)
- `GET /workspaces/{id}` (детали)
- `GET /nav/views` (каталог view/route) — можно статично, но лучше через API
- `GET /commands` (команды для Ctrl+K)
