# IFOS 37601–38000 — Agents & Workflow Automation OS (v1)
Цель: сделать “интернет функций” **исполняемым**:
- у каждой функции есть описание (tool descriptor)
- функции собраны в реестр (tool registry)
- агент умеет планировать (plan) и выполнять (run)
- опасные действия требуют подтверждения (approvals)
- есть политики (что можно/нельзя)
- есть мост к Make/n8n (bridge)
- всё логируется (audit), измеряется (observability)

Порядок: простое → среднее → сложное.

---

## 37601–37630 — AgentProfile: кто выполняет действия
AgentProfile = “роль агента”:
- цели (что оптимизируем: скорость/качество/стоимость)
- доступы (какие реестры, какие тенанты)
- предпочтения (язык, стиль отчётов)
- ограничения (не трогать финансы, не писать во внешние системы без approval)

Это решает хаос: “ИИ сделал не то”.

---

## 37631–37680 — ToolDescriptor: описание одной функции
ToolDescriptor — карточка инструмента:
- входы/выходы (schema)
- side effects (пишет ли куда-то)
- уровень риска (safe / confirm / restricted)
- rate limits / costs
- required secrets
- примеры вызова

ToolDescriptor превращает “скрипт” в **продукт**.

---

## 37681–37710 — ToolRegistry: каталог функций
ToolRegistry хранит:
- набор ToolDescriptor
- версии
- “пакеты инструментов” (bundles) под сценарии: CRM, HR, Helpdesk, Marketing
- совместимость (dependencies)

Идея: “установить набор инструментов как приложение”.

---

## 37711–37750 — WorkflowSpec: сценарий из шагов
WorkflowSpec = “Make-сценарий, но универсальный”.
Содержит:
- шаги (steps)
- зависимости (depends_on)
- условия (if/else)
- циклы (for each)
- обработку ошибок (retry, fallback)
- переменные/контекст

Главное: workflow описан декларативно и переносим.

---

## 37751–37790 — Tasks & Plans: от задач к плану
### TaskItem
Задача — атомарная единица работы (ticket).
TaskItem включает:
- описание
- входные данные
- ожидания результата (definition of done)

### PlanSpec
План — разложение на шаги + проверяемые checkpoints:
- что делаем
- чем делаем (tools)
- как проверяем результат
- где нужен approval

Это позволяет делать “простые макросы” и “сложные кампании”.

---

## 37791–37830 — ExecutionRun: запуск и состояние
ExecutionRun хранит:
- статус (queued/running/succeeded/failed)
- шаги и их результаты
- артефакты (файлы, ссылки, данные)
- retries/backoff
- idempotency keys (чтобы не сделать дважды)

Так появляется “кнопка запуск/повтор”.

---

## 37831–37870 — Approvals: человек в контуре
ApprovalRequest создаётся когда:
- действие финансовое/юридическое
- отправка писем/сообщений наружу
- удаление/перезапись данных
- доступ к секретам

ApprovalRequest содержит:
- что будет сделано
- аргументы (payload)
- риски
- expected outcome
- кто может approve

---

## 37871–37910 — ActionPolicy: правила безопасности
ActionPolicy описывает:
- allow/deny по tool, по scope, по данным
- ограничение по времени (не ночью)
- лимиты (не более N писем)
- redaction (не передавать PII)
- require citations (для фактов)

Policy-as-code = управляемые “перила”.

---

## 37911–37950 — Bridge: Make/n8n/Node-RED как исполнители
ConnectorBridge связывает IFOS с:
- Make.com (сценарий по webhook)
- n8n (workflow by API)
- Zapier / Pipedream
- self-hosted Node-RED

Bridge решает задачу: “есть 1000 готовых сценариев — сделаем их частью IFOS”.

---

## 37951–37980 — AuditEvent: журнал действий
AuditEvent хранит:
- кто запустил
- какие инструменты вызваны
- какой результат
- какие данные затронуты
- какие approvals были

Это обязательная база для enterprise.

---

## 37981–38000 — Observability: метрики и здоровье
Минимум:
- success rate по workflow
- latency по шагам
- cost per run
- top failing tools
- approval bottlenecks

Это делает систему поддерживаемой.
