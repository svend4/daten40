# IFOS 38401–38800 — UI Builder & Interaction OS (v1)
Цель: превратить “интернет функций” в **понятный пользовательский интерфейс**, а не в список API.
Здесь появляется:
- UI Cards (карточки функций/пакетов/сценариев)
- Vitrines (витрины: коллекции карточек, как “магазин”)
- Forms Builder (конструктор форм и мастеров)
- Interaction Modes (ручной / полуавто / авто)
- Explainability UI (почему так, откуда данные, какие риски)
- A11y (доступность)
- i18n (локализация)
- Telemetry (события UI, аналитика без шпионажа)

Порядок: простое → среднее → сложное.

---

## 38401–38430 — UI Card: единица интерфейса
UI Card = минимальный “виджет-объект”:
- заголовок, иконка, теги
- краткое описание “что делает”
- входы (какие параметры просит)
- действия (кнопки: запустить, посмотреть примеры)
- уровень риска (safe/confirm/restricted)
- ссылки на документацию/отзывы/версии

Это “как приложение в магазине”, только для функций.

---

## 38431–38470 — Vitrine Builder: витрины и коллекции
Vitrine = подборка карточек под задачу:
- “Автоматизация офиса 2.0”
- “CRM для малого проекта”
- “Импорт CSV → дедуп → реестр”
- “Travel Hub: сравнение + партнёрки”

Витрина хранит:
- порядок карточек
- фильтры/категории
- рекомендованные связки (bundles)
- onboarding: “с чего начать”

---

## 38471–38520 — Forms Builder: мастера вместо настроек
UIForm = декларативная форма:
- поля (text/select/file/date)
- валидация (regex, required)
- подсказки (help)
- шаблоны (prefill)
- шаги (wizard) и зависимые поля

Смысл: пользователь не видит “JSON payload”, он видит понятную форму.

---

## 38521–38560 — UI State Model: состояние и контекст
UIStateModel описывает:
- текущий контекст (project, tenant, bundle)
- выбранные карточки/фильтры
- черновики форм
- история действий
- ссылки на execution runs

Это база для “одной кнопки повторить” и “вернуться к шагу 2”.

---

## 38561–38610 — Interaction Modes: ручной/полуавто/авто
InteractionMode:
- Manual: пользователь нажимает каждый шаг
- Assisted: агент предлагает план, пользователь подтверждает
- Auto: агент выполняет, но с approvals на опасные шаги

Смысл: одна и та же витрина работает для новичка и для профи.

---

## 38611–38650 — Explainability Bundle: прозрачность
ExplainabilityBundle показывает:
- какие источники использованы (citations)
- какие правила сработали (policies)
- почему выбран такой инструмент (ranking reasons)
- риски и что будет изменено
- “как отменить” (rollback hint)

Без explainability интерфейс превращается в “магический ящик”.

---

## 38651–38700 — UX Policy: “перила” в интерфейсе
UXPolicy задаёт:
- что показывать/скрывать по ролям
- какие кнопки требуют двойного подтверждения
- предупреждения (PII, финансовые действия)
- режим обучения (tooltips)
- safe defaults

UXPolicy = безопасность на уровне UI, а не только backend.

---

## 38701–38740 — Accessibility (A11y): интерфейс для всех
AccessibilityProfile:
- крупные шрифты / контраст
- управление клавиатурой
- screen reader labels
- понятные ошибки
- упрощённые формы

Это критично для гос/мед/соц проектов и для реального “Office 2.0”.

---

## 38741–38770 — Localization (i18n): много языков
LocalizationPack:
- строки UI
- форматы дат/валют
- терминология (“реестр/registry”, “витрина/vitrine”)
- глоссарий (согласованные переводы)

Иначе система будет “ломаться” на переводах.

---

## 38771–38800 — UI Telemetry: метрики без слежки
UIEvent фиксирует:
- клики/ошибки/время прохождения мастера
- падения шагов (связь с run)
- где пользователи застревают

Принцип: telemetry = улучшение продукта, не рекламный трекинг.

## Итог блока
UI Builder & Interaction OS делает IFOS понятным:

- формы вместо JSON

- explainability + a11y + i18n
- минимальная аналитика для улучшения
