# IFOS 18401–18800 — Knowledge & Documentation‑OS: унификация документации, авто‑мануалы из схем/логов, интерактивные FAQ, “объясни этот workflow”, глоссарий, генерация туториалов из успешных запусков (v1)

Вы хотите “не искать новое”, а **рационализировать**: взять миллионы решений и сделать их **понятными и собираемыми в кластеры**.  
Это невозможно без **единого слоя знаний**: одинаковая структура документации, одинаковые карточки “как работает”, и автоматика — чтобы новые модули сразу становились объяснимыми.

Связи:
- Registry‑OS: что это за модуль
- UX & One‑Click‑OS: как человек нажимает кнопки
- Trust & Compliance‑OS: можно ли этому доверять
- **Knowledge & Documentation‑OS: как это понять, освоить и чинить**

Ниже — по порядку, от простого к сложному.

---

## 18401–18460 — Doc Cards & Doc Pages (простое)

### 18401) Doc Card = “краткий смысл”
Doc‑Card — компактная “памятка” для ассета (плагина/рецепта/адаптера):
- что делает
- входы/выходы
- примеры “до/после”
- требования (аккаунты, токены, права)
- типичные ошибки + быстрые решения
- ссылки на туториалы/FAQ

Doc‑Card должен помещаться в 1 экран и дублировать ключевое из UI‑карточки, но **в формате знаний**.

### 18420) Doc Page = “полная страница”
Единый шаблон страницы:
1) Overview (что/зачем)  
2) Quickstart (5 шагов)  
3) Inputs/Outputs (таблица)  
4) Install/Run (варианты: Make/n8n/WP/native)  
5) Troubleshooting (частые ошибки)  
6) Security/Privacy notes (что логируется, где секреты)  
7) Changelog  
8) References (registry ids, schemas, evidence)

**Принцип:** любые 2 ассета читаются одинаково.

## 18461–18520 — Glossary (среднее): единые слова вместо хаоса терминов

### 18461) Термин как объект (не текст)
Термины должны быть:
- определение простыми словами
- техническое определение
- примеры
- “похожие термины” (webhook vs polling)
- ссылка на ассеты/страницы, где употребляется

Это решает проблему “не рекламируются / не прославлены”: термин связывает всё в одну карту.

## 18521–18580 — Interactive FAQ (среднее): ответы, которые ведут к действиям

### 18521) FAQ item должен иметь “кнопки”
FAQ не просто текст. У каждого ответа:
- “показать в wizard” (перейти на шаг)
- “проверить соединение” (test connection)
- “показать пример конфигурации”
- “создать тикет / issue” (если баг)

### 18540) FAQ с доказательностью
FAQ должен ссылаться:
- на run evidence (“мы это видели в логах”)
- на troubleshooting case id
- на policy decisions (если запрет)

## 18581–18640 — Troubleshooting cases (среднее+): база ошибок как “каталог болезней”

### 18581) Case = ошибка → причины → проверки → фиксы
Стандартизированная карточка “случай”:
- симптомы
- вероятные причины (по вероятности)
- проверки (команды/кнопки)
- решение (шаги)
- профилактика (как не повторять)
- ссылки на логи (anonymized)
- версии, где исправлено

### 18610) Ошибки как данные
Когда у вас тысячи модулей, **ошибки — это главное знание**.  
По сути, это “медицина для интернет‑модулей”.

## 18641–18700 — Auto‑Docs from Schemas + Logs (сложнее): документация сама себя пишет

### 18641) Генерация из схем (inputs/outputs)
Если у ассета есть JSON‑schema или OpenAPI:
- автоматически строим таблицы полей
- автогенерируем формы ввода
- автогенерируем примеры payload

### 18670) Генерация из логов/успешных runs
Из успешных запусков:
- вытаскиваем реальные параметры (с редактированием PII)
- превращаем в пример конфигурации
- собираем “шаги” как туториал

Это ключ к вашей идее “миллионы решений”: документация масштабируется **сама**.

## 18701–18760 — “Explain this workflow” (сложно): объяснение связок Make/n8n/WP

### 18701) Объяснение workflow по шагам
Система должна уметь:
- прочитать workflow (Make scenario / n8n json / WP plugin config / native yaml)
- выделить блоки (триггеры, трансформы, действия)
- объяснить “что куда течёт”
- показать карту данных (dataflow)
- подсветить риски (где секреты, где PII, где внешние вызовы)
- предложить улучшения (dedup, retries, backoff)

### 18730) Explain output = пригодно для человека и для машины
Результат объяснения — объект, который:
- отображается в UI
- индексируется поиском
- прикрепляется к ассету как “понимание связки”

## 18761–18800 — Knowledge Graph + Doc Bundles (самое сложное): знания как система

### 18761) Knowledge Graph (кто с кем связан)
Граф связывает:
- ассеты (bundles, adapters, recipes)
- термины (glossary)
- FAQ
- troubleshooting cases
- туториалы
- evidence runs

**Итог:** “прославление” становится навигацией: лучший модуль видно в графе по связям и доказательствам.

### 18790) Doc Bundle export (для офлайна и архивов)
Экспортируем пакет:
- страницы (md/html)
- карточки
- глоссарий
- FAQ
- troubleshooting
- индексы поиска
- связи графа

Это позволяет хранить “интернет‑операционку” локально, в wiki/Git/архиве.
