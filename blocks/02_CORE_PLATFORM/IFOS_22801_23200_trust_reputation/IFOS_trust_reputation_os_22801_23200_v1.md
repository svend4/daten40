# IFOS 22801–23200 — Trust & Reputation OS: вертикаль качества, верификация, trust signals, отзывы/рейтинги, анти‑спам/анти‑накрутка, доказательства (v1)

Вы сформулировали главную боль “дикого интернета”:
- миллионы плагинов/шаблонов/сценариев/скриптов есть,
- но **нет вертикали качества**: не ясно, что надёжно, что совместимо, что безопасно,
- нет нормальной системы отзывов и доказательств,
- нет “прославления по уровню”, как у стандартных офисных функций.

Этот блок делает **слой доверия** поверх каталога функций/кластеров/пакетов:
- фиксирует **уровни качества L0–L5**,
- собирает **trust signals** (факты + данные наблюдаемости),
- добавляет **верификацию (подписи, SBOM)**,
- строит **рейтинги + анти‑спам/анти‑накрутку**,
- и даёт **политику**: что можно ставить “в один клик” на прод, а что только в sandbox.

Дальше — по порядку, от простого к сложному.

## 22801–22870 — Лестница качества L0–L5 (простое, но фундаментальное)

### 22801) Зачем нужна “лестница качества”
Без лестницы всё равнозначно: хороший пакет и мусор стоят рядом.
Лестница — это “вертикаль власти” качества, как у стандартов.

### 22810) Предлагаемая шкала (минимум)
**L0 — Draft**
- нет тестов, нет подписи, неизвестный автор
- можно только “посмотреть”, нельзя ставить на прод

**L1 — Basic**
- есть описание, зависимости, примеры
- можно ставить в dev/sandbox

**L2 — Verified**
- есть smoke tests + receipt
- есть базовые trust signals (успешные установки, низкий fail rate)

**L3 — Reliable**
- есть SLO‑метрики из Observability OS
- есть rollback plan, миграции, стабильность версий

**L4 — Audited**
- есть подпись publisher’а + SBOM
- проверены лицензии и политики
- есть независимые доказательства (evidence), минимум N реальных пользователей/инсталлов

**L5 — Enterprise/Gov**
- строгие policy gates
- оффлайн/закрытый контур поддержан
- формальные отчёты/сертификаты, контроль изменений, “2‑person review”

> Важно: уровень — не “мнение”, а **набор выполняемых требований**.

## 22871–22950 — Trust signals: факты, которые можно измерить

### 22871) Что такое TrustSignal
TrustSignal — атом доверия. Он должен быть:
- измеряемым или проверяемым,
- привязанным к субъекту (asset/bundle/publisher),
- иметь источник и дату.

Типы сигналов:
- **Coverage**: есть docs/tests/runbooks
- **Stability**: crash rate, fail ratio, MTTR
- **Security**: подписан, SBOM есть, secrets не логируются
- **Adoption**: установки, активные инстансы
- **Support**: время ответа, закрытие инцидентов
- **Compatibility**: compat score по платформам (Make/n8n/WP)

### 22910) Как trust signals связываются с Observability OS
Часть сигналов напрямую берётся из 22401–22800:
- fail_ratio, delivery_success_ratio, burn_rate SLO
- инциденты и постмортемы
- частота rollback

## 22951–23030 — Верификация: подписи, SBOM, provenance (среднее → сложное)

### 22951) Подпись пакета (SignatureEnvelope)
Цель: гарантировать, что:
- пакет не подменён,
- известен издатель (publisher),
- версия и хэши совпадают.

### 22970) SBOM (Software Bill of Materials)
Для кода и контейнеров — список компонентов и лицензий.
Для no‑code — список модулей/коннекторов/permissions.

### 22990) Provenance chain (цепочка происхождения)
Откуда пришло:
- репозиторий, коммит, сборка,
- кто опубликовал,
- какие проверки прошёл.

## 23031–23110 — Отзывы и рейтинги: но не “как в магазине”, а инженерно

### 23031) Review = структурированная оценка
Отзыв должен иметь:
- контекст (где ставили: платформа/версия/профиль)
- результат (успех/неуспех)
- наблюдаемость (ссылки на receipt/logs)
- полезные поля: “что пошло не так”, “как исправили”, “стоимость”

### 23060) RatingAggregate
Не один “средний балл”, а набор:
- reliability score
- docs score
- support score
- value/cost score
- risk score (обратная метрика)

### 23090) Ранжирование отзывов
Отзыв “пустой” без доказательств — ниже.
Отзыв с receipt + trace + конкретикой — выше.

## 23111–23170 — Анти‑спам, анти‑накрутка, борьба с “инфошумом”

### 23111) AntiSpamRule
Правила простого уровня:
- ограничение частоты отзывов
- запрет дубликатов
- требования к минимальной структуре

### 23135) AntiFraudDetection
Эвристики:
- кластер новых аккаунтов с одинаковыми фразами
- внезапный всплеск 5★ без инсталлов
- конфликт signals: “всё отлично” при high fail ratio

**Ключ:** сопоставление *слов* и *телеметрии*.

## 23171–23200 — Trust policy: что можно “в один клик”

### 23171) TrustPolicy
Политика связывает:
- минимальный уровень качества (например, L2 для sandbox, L3 для prod)
- требования к подписи/издателю
- требования к SLO/инцидентам
- запреты по лицензиям и PII

### 23190) Appeals / governance
Если пакет пометили “risk” — должен быть процесс апелляции:
- какие доказательства нужны
- кто рассматривает
- какие сроки и уровни
