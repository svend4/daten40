# IFOS 28001–28400 — Security & Governance OS (v1)
Этот блок отвечает за то, чтобы “операционная система функций интернета” была **управляемой, безопасной и проверяемой**.
Если у вас есть тысячи интеграций/плагинов/сценариев, то без Governance вы получите:
- утечки секретов,
- случайные публикации/платежи,
- невозможность понять “кто что сделал”,
- хаос прав и ответственности,
- недоверие (невозможно повторить результат и доказать источники).

Ниже — **по порядку, от простого к сложному**.

---

## 28001–28030 — Data Classification: классы данных (простая табличка, которая решает 80%)
Классы:
- PUBLIC (можно публиковать)
- INTERNAL (внутреннее)
- CONFIDENTIAL (чувствительные данные: токены, финансы)
- PERSONAL (персональные данные)
- RESTRICTED (строго ограничено, минимальный доступ)

Каждая коллекция/артефакт/лог получают метку класса данных.
Это автоматически включает правила:
- где хранить,
- кто может видеть,
- сколько хранить (retention),
- можно ли экспортировать.

---

## 28031–28070 — Secrets: секреты как ссылки, а не “текст в настройках”
**Правило №1:** в записях/пакетах/flows секретов нет. Есть только `secret_ref`.
Secrets OS:
- SecretRef: “ссылка на секрет” (имя + область + тип)
- SecretStore: где секрет реально хранится (локально/крипто‑хранилище/облачный KMS)
- Ротация: смена токена без изменения flows
- Least privilege: токен только с нужными правами

Результат: вы можете безопасно делиться пакетами и шаблонами.

---

## 28071–28130 — Roles/ACL: роли, права, привязки (минимально, но строго)
Минимальный набор ролей:
- Owner (владелец workspace)
- Admin (управление настройками/инфраструктурой)
- Builder (создаёт flows/пакеты, но не видит секреты)
- Operator (запускает jobs, смотрит статусы)
- Viewer (только чтение витрин/отчётов)

ACL Binding:
- кто (user/service)
- что (resource: collection/job/package/secret)
- какие действия (read/write/run/admin)

Важно: “право запускать” ≠ “право менять” ≠ “право видеть секрет”.

## 28131–28190 — Policies: политики безопасности как конфиг
Policy — машино‑читаемое правило:
- allow/deny по действиям
- условия (по роли, по времени, по IP/региону, по классификации данных)
- обязательные меры: 2FA, approvals, signed packages
- лимиты: max exports/day, max runs/hour

Примеры политик:
- “Экспорт PERSONAL данных запрещён без approval”
- “Публикация витрины только из TRUSTED источников”
- “Включить audit для всех run’ов в продакшене”

---

## 28191–28240 — Audit: журнал действий (кто/что/когда/почему)
AuditEvent фиксирует:
- actor (user/service)
- action (create_job, run_job, export_bundle, add_secret_ref…)
- resource refs
- результат (success/fail)
- контекст (ip, user_agent, correlation_id)

Без audit нельзя управлять системой “как бизнес‑ОС”.

---

## 28241–28290 — Moderation: модерация данных и пакетов (контент и качество)
Moderation нужна не “ради цензуры”, а ради качества и безопасности:
- блокировать вредоносные/паразитные источники
- отсекать спам/мусор
- помечать сомнительное (needs_review)
- правила для “авто‑публикаций” (нельзя публиковать без проверки)

ModerationDecision:
- объект (record/artifact/package/source)
- решение: allow / block / quarantine / needs_review
- причины (policy_violation, malware_risk, low_trust, pii_detected…)
- кто принял (human/ai) и какой confidence

---

## 28291–28330 — Trust Lists: доверенные источники и поставщики
TrustList:
- домены/организации/паблишеры
- репозитории/пакеты (supply chain)
- критерии доверия (подпись, репутация, ручная проверка)

Это отвечает на ваш вопрос “почему интернет — дикое поле”:
IFOS формирует **слои доверия**, а не “всё подряд из сети”.

---

## 28331–28370 — Package Signatures: подписи и проверка пакетов (Supply Chain Security)
Пакеты (bundles) должны:
- иметь manifest + checksums (из Data OS)
- иметь signature (кто выпустил пакет)
- проходить verify перед установкой/запуском

PackageSignature:
- signer_id
- algo (ed25519/ecdsa)
- signature over manifest hash
- timestamp + optional transparency log

Результат: “поставили чужой пакет” ≠ “поймали вирус”.

---

## 28371–28400 — Compliance Profile + Consent: соответствие правилам и согласия
ComplianceProfile задаёт набор требований:
- хранение персональных данных (минимизация, сроки)
- экспорт/удаление (right to delete)
- журналирование (audit mandatory)
- шифрование (at rest / in transit)

ConsentRecord:
- кто дал согласие (user/customer)
- на что (категории данных/цели)
- срок
- отзыв согласия

Важно: даже без юридической глубины это делает систему “взрослой”.
