# IFOS 27201–27600 — Automation & Scheduling OS (v1)
Этот блок превращает “кнопки” и “карточки” в **регулярно работающую систему**: по расписанию, по триггерам, с очередями, SLA, ретраями, мониторингом и алертами.

Ниже — **по порядку, от простого к сложному**.

---

## 27201–27230 — Schedule: расписание как объект (cron/rrule/interval)
Schedule — единый формат расписаний:
- cron (*/15 * * * *)
- rrule (iCal‑стиль, полезно для бизнес‑графиков)
- interval (“каждые 10 минут”)
- timezone (важно!)

Зачем: один и тот же механизм для Make‑сценариев, WordPress‑задач, локальных задач и серверных job’ов.

---

## 27231–27270 — Trigger: события, которые запускают работу
Trigger типы:
- manual (кнопка)
- schedule (по расписанию)
- webhook (входящий HTTP)
- event (из телеметрии/аудита)
- file_drop (появился файл)

Trigger связывает:
- что слушаем
- какие фильтры
- какую Job запускаем

---

## 27271–27320 — Job: единица автоматизации
Job = “запуск потока/макроса/пакета” с параметрами:
- job_id, name, workspace_id
- target: flow_id / package_id / macro_batch_id
- input bindings (подставить переменные из формы/секретов/контекста)
- schedule/trigger refs
- retry_policy_ref, sla_policy_ref, rate_limit_ref
- idempotency_key strategy

Job — это “скрипт” уровня бизнеса, не привязанный к конкретной платформе.

---

## 27321–27380 — Queue + Worker: очередь и исполнители (надёжность)
Queue:
- priority (high/normal/low)
- concurrency (сколько параллельно)
- visibility timeout (защита от “повисших” задач)
- dead‑letter queue (куда падают безнадёжные)

Worker:
- capabilities (что умеет исполнять: wordpress/make/fastapi/local)
- resource limits
- heartbeat

Смысл: без очередей любые “планы на каждый день” превращаются в хаос.

---

## 27381–27430 — RetryPolicy + Idempotency: ретраи без дубликатов
RetryPolicy:
- max_attempts
- backoff: linear/exponential
- jitter
- retry_on: codes (429/500/network)

IdempotencyKey:
- ключ = (job_id + входы + дата‑окно)
- хранение ключей + TTL
- правило: если уже выполнено — не повторять

Это критично для платежей/экспорта/публикаций/уведомлений.

---

## 27431–27480 — SLA Policy + RateLimit: бизнес‑гарантии и лимиты
SLA Policy:
- максимальная задержка старта (например ≤5 минут)
- максимальная длительность (например ≤2 минуты)
- окна обслуживания (ночью можно дольше)

RateLimit:
- ограничение запросов к API (например 60/min)
- per‑tenant / per‑workspace

Зачем: система должна “вести себя прилично” и не баниться у провайдеров.

---

## 27481–27530 — Monitoring & Alerting: наблюдение и реакция
MonitorRule:
- условие (ошибок > N за T)
- “нет успешных запусков за 24 часа”
- SLA breach (превышена задержка)
- рост времени ответа

Alert:
- канал: email/telegram/webhook
- дедуп алертов (не спамить)
- escalation (если не ответили — кому дальше)

---

## 27531–27580 — Cron Library: библиотека “типовых расписаний”
Нужно не “каждый раз писать cron”, а выбирать шаблон:
- “каждый день 09:00”
- “каждый понедельник 08:30”
- “каждый час”
- “первое число месяца”
- “рабочие дни (пн‑пт)”

CronLibrary хранит готовые правила с человеко‑понятными названиями.

---

## 27581–27600 — MacroBatch: пакетные макросы “одна кнопка — регулярно”
MacroBatch — сценарий из макросов/flow‑фрагментов:
- “Собрать новости → сделать бриф → отправить в Telegram → сохранить в архив”
- “Импорт отзывов → нормализация → рейтинг → публикация витрины”

MacroBatch связывает UI‑уровень и автоматизацию: пользователь понимает “что будет происходить каждый день”.

---

## Мини‑архитектура Automation OS
1) Job описывает “что делать”  
2) Trigger/Schedule решает “когда запускать”  
3) Queue/Worker обеспечивает “как исполнять надёжно”  
4) Retry/Idempotency не даёт сломать данные  
5) SLA/RateLimit держит дисциплину  
6) Monitoring/Alerting делает систему управляемой
