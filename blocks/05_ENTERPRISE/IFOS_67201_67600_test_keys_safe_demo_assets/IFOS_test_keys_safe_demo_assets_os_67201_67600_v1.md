# IFOS 67201–67600 — Test Keys & Safe Demo Assets OS (v1)
Цель: закрыть боль разработчиков и компаний: **где взять тестовые ключи/песочницы/демо‑данные**, чтобы:
- не “сливать” реальные токены и PII
- быстро запускать интеграции (Make/n8n/WordPress плагины/SDK)
- делать витрины, демо и обучение “как Office” — один клик, понятные шаблоны

Этот блок даёт 4 результата:
1) **Каталог песочниц (Sandbox Providers)** по сервисам
2) **Каталог тестовых ключей (Test Key Catalog)** + политики безопасности
3) **Safe Demo Assets** (демо‑объекты: фейковые CRM контакты, фейковые заказы, “псевдо‑инвойсы”)
4) **One‑Click Trial Bundles** (наборы: коннектор + демо‑данные + макросы + инструкции)

Связи:
- использует **Credential Vault** (61601–62000) и **Identity/Access**
- запрещает утечки через **DLP & Classification** (66401–66800)
- выдаёт receipts в **Forensics** (66001–66400)

Порядок: простое → среднее → сложное.

---

## 67201–67240 — Простое: единые понятия
### SandboxProvider
Описание тестовой среды конкретного SaaS/API:
- где включается (UI/URL)
- какие ограничения (rate limits, allowed IPs, data retention)
- какие тестовые события доступны (webhooks test events)
- какие “демо‑объекты” можно создать автоматически

### TestKeyEntry
Запись в каталоге тестовых ключей:
- сервис (provider), продукт (api scope)
- тип ключа (api key / oauth client / webhook secret / token)
- как получить (самостоятельно/по заявке/генерируется системой)
- требования: sandbox only, scopes, region, expiry

---

## 67241–67320 — База: политики ключей (CredentialPolicy)
CredentialPolicy определяет:
- где хранить (vault namespace)
- ротация (каждые N дней)
- минимальные scopes
- запрет прод‑эндпоинтов
- allowlist callback URLs (для OAuth)
- запрет передачи ключей в лог/тикеты

Золотое правило:
> любой “тестовый ключ” должен быть технически **непригоден** для прод‑данных.

---

## 67321–67400 — Среднее: KeyRequest → KeyGrant
### KeyRequest (заявка)
- кто просит (команда/тенант/разработчик)
- зачем (purpose)
- какие интеграции (connector ids)
- какие scopes нужны (минимум)
- на сколько (ttl)

### KeyGrant (выдача)
- какая политика применена
- срок действия и ротация
- ограничения (rate, ip allowlist)
- “canary tag” для последующего расследования

---

## 67401–67480 — Среднее+: Safe Demo Assets
DemoAsset — безопасные объекты для демо и тестов:
- “фейковые” контакты, компании, сделки, счета, тикеты
- наборы событий (webhook fixtures)
- mock files (pdf invoices) без персональных данных
- synthetic datasets (ссылка на 66801–67200)

DemoAsset должен иметь:
- класс данных PUBLIC/INTERNAL (без PII)
- генератор (скрипт/макрос/Make scenario)
- resets (как быстро “обнулить” демо‑аккаунт)

---

## 67481–67560 — Сложное: One‑Click Trial Bundles
TrialBundle — пакет “запусти интеграцию за 10 минут”:
- connectors list + версии
- demo assets + safe datasets
- готовые макросы (workflow runner)
- инструкции (steps)
- checks (smoke tests)

Пример: “Stripe Sandbox + CRM Demo + Invoice PDF Generator”
- создаёт тестовые платежи
- отправляет их в CRM
- генерирует PDF инвойс (без PII)
- показывает витрину результатов

---

## 67561–67600 — Сложное+: Авто‑проверки и анти‑утечки
Минимальный “guardrail pipeline”:
1) Проверка endpoint: sandbox домены/окружение
2) Проверка scopes: denylist опасных
3) DLP scan на случай, если кто-то вставил “реальный” ключ
4) Логи: redaction секретов, запрет вывода в UI
5) Rotation jobs: ключи не живут вечно
