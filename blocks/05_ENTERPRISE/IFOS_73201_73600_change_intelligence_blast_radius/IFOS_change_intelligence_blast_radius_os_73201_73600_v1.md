# IFOS · Change Intelligence & Blast Radius (Release/Change Impact Engine)
**Диапазон:** 73201–73600  
**Slug:** `change_intelligence_blast_radius` · **Версия:** 0.33.0 · **Статус:** DONE

## 1) Зачем нужен этот блок
Интернет и корпоративные ИТ-системы уже наполнены готовыми компонентами (плагины, коннекторы, SaaS, SDK, готовый код), но «управление изменениями» в этом мире часто остаётся ручным: релизы, конфиги, фичи-флаги и миграции происходят постоянно, а связь «изменение → эффект» теряется.

Этот блок добавляет *Change Intelligence* — единый слой, который:
- собирает все изменения (релизы/деплои/конфиги/фичи/инфра) в **единую ленту (Change Timeline)**;
- связывает изменения с сервисами/владельцами по топологии (**Service Catalog + Topology Graph**);
- коррелирует изменения с метриками/логами/трейсами/инцидентами (**AIOps/Observability**);
- оценивает **blast radius** (радиус поражения/влияния) до и после выката;
- формирует объяснимые рекомендации: «почему деградировало», «что откатить», «какая команда отвечает».

## 2) Что входит (функции)
### 2.1 Change Timeline (единая лента изменений)
- агрегация событий: `git commit/tag/release`, CI/CD deploy, feature-flag toggle, config change, infra change, DB миграции;
- нормализация в общий формат **ChangeEvent**;
- фильтры: по сервису/домену/команде/окружению/времени;
- дифф/артефакты изменения (ссылки на PR, pipeline, image digest, config diff).

### 2.2 Change ↔ Observability Correlation
- связывание ChangeEvent с:
  - аномалиями (AIOps),
  - нарушениями SLO/SLI,
  - инцидентами (IR),
  - синтетическими сценариями и RUM,
  - регрессиями производительности.
- выдача «объяснимых» причинно-следственных карточек:
  - **Evidence**: какие метрики изменились, какой trace-path вырос, какие пользователи затронуты.
  - **Confidence**: уверенность корреляции.

### 2.3 Blast Radius Estimation (оценка радиуса влияния)
Оценка влияния выполняется по топологии и сигналам нагрузки:
- *топологическое распространение*: upstream/downstream зависимости;
- *вес критичности сервиса*: tier, бизнес-критичность, SLO-класс;
- *нагрузка/трафик*: доля запросов, RUM-сессии, synthetic-journeys;
- *контекст выката*: progressive delivery, фичи-флаг, аудитория, регионы;
- *исторические паттерны*: «похожие изменения раньше».

Результат: **ImpactReport** с ранжированием сервисов/journey/сегментов пользователей и рекомендациями (hold/rollback/continue).

### 2.4 Guardrails и рекомендации по откату
- правила «не продолжать» при нарушении SLO/ошибках/росте латентности;
- рекомендации: откат релиза, выключение фичи, rollback миграции (если возможно), откат конфигов;
- интеграция с progressive delivery и feature flags (опционально).

### 2.5 Аннотации (Observability UX)
- автоматическое добавление аннотаций на графики и дашборды по ключевым ChangeEvent;
- «скачки» на таймлайне: релизы, инциденты, откаты.

## 3) Не входит (границы блока)
- полный CI/CD и управление пайплайнами (это отдельные блоки/платформы);
- глубокая безопасность поставки (SLSA/SBOM/подписи) — может быть подключено как зависимость;
- управление исходным кодом и ревью — интеграция через ссылки/вебхуки.

## 4) Ключевые сущности данных
### 4.1 ChangeEvent (нормализованное событие изменения)
Минимальные поля:
- `change_event_id`, `timestamp`, `environment`, `source_system`
- `type` (deploy/release/flag/config/infra/db_migration/…)
- `service_ids[]` (после маппинга)
- `artifact_refs[]` (git sha, image digest, package version)
- `actor` (user/service account), `links[]`, `diff_ref`

### 4.2 ImpactEdge (Change → Service/Node)
- `change_event_id`, `topology_node_id`, `impact_score`, `confidence`
- `evidence_refs[]` (метрики, трейсы, тесты, алерты)
- `propagation_path[]` (как влияние «прошло» по топологии)

### 4.3 ImpactReport
- окно времени, список затронутых узлов/сервисов с оценкой
- рекомендации (hold/rollback/continue)
- объяснение (top evidences)

## 5) Пайплайн обработки (упрощённо)
1. **Ingest**: принять события из источников (вебхуки/пуллинг).
2. **Normalize**: привести к ChangeEvent.
3. **Enrich**: сопоставить артефакты и сервисы (через Service Catalog / mapping matrix).
4. **Correlate**: привязать к аномалиям/инцидентам/метрикам.
5. **Estimate**: посчитать blast radius по правилам и графу.
6. **Publish**: отправить в UI/Feed/Аннотации/уведомления.

## 6) API (черновой контур)
- `POST /change-events` — принять change events (bulk/stream)
- `GET /change-events` — поиск по фильтрам
- `POST /blast-radius/estimate` — запрос оценки влияния (перед выкатыванием)
- `GET /blast-radius/reports/{id}` — получить отчёт
- `POST /observability/annotations` — создать аннотации в мониторинге

## 7) UX-экран (минимально)
- **Change Feed**: лента изменений + фильтры
- **Change Details**: артефакты, ссылки, diff, владельцы
- **Impact Map**: список сервисов и мини-граф распространения
- **Why this?**: объяснимые evidence-карточки
- **Actions**: rollback/disable flag/hold rollout (если подключены интеграции)

## 8) Seed-данные (что даём в комплекте)
- типы событий и обязательные поля;
- правила/веса для blast radius;
- каталог источников (какие коннекторы нужны, формат событий, auth).

## 9) MVP (P0/P1) и эволюция
**P0:** ручной импорт change events + аннотации + базовый blast radius по топологии.  
**P1:** корреляция с инцидентами/SLO + простая рекомендация rollback.  
**P2:** auto-mapping артефактов к сервисам + confidence/evidence.  
**P3:** обучение на исторических данных (паттерны «подобных изменений»).
