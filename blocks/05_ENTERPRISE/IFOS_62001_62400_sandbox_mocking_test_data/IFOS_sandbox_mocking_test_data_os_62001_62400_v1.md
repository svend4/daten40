# IFOS 62001–62400 — Sandbox, Mocking & Test Data OS (v1)
Цель: убрать главную боль интеграций и обучения: **нет тестовых данных, нет ключей, нет повторяемых сценариев**.  
Этот блок делает “песочницу” стандартной частью IFOS: любой bundle/connector получает воспроизводимую среду для демо и тестов.

Порядок: **простое → среднее → сложное**.

---

## 62001–62020 — Базовые понятия
**Sandbox** (песочница) = безопасная среда, где можно:
- запускать интеграции без реальных денег/клиентов/рисков
- проверять коннекторы (health checks)
- демонстрировать “one‑click” bundle
- генерировать и подставлять тестовые данные

**Fixture** = отдельный тестовый объект (пример: “контакт”, “заказ”, “письмо”).  
**Dataset** = набор fixtures + правила связей.  
**Generator** = фабрика данных (генерирует fixtures с параметрами).  
**Mock server** = имитация API, если реального sandbox нет или он нестабилен.  
**Record/Replay** = запись реального трафика и воспроизведение в песочнице.

---

## 62021–62070 — Быстрый путь “одна кнопка” для пользователя
UX поток:
1) Установил bundle
2) IFOS предлагает: **Start in Sandbox**
3) Выбор: (A) Official sandbox (если есть) / (B) Mock server / (C) Replay recording
4) IFOS создаёт Dataset (fixtures) + поднимает mock/proxy
5) Запуск теста → зелёные health checks
6) При желании “Switch to Prod” (и уже подключить настоящие ключи/данные)

Результат: интеграции становятся “как офисные макросы” — воспроизводимо и понятно.

---

## 62071–62120 — Catalog песочниц (Sandbox Catalog)
Для каждого коннектора IFOS хранит:
- поддерживаемые режимы: official sandbox / mock / replay
- ссылки на документацию (как получить test mode)
- лимиты и reset policy
- известные “подводные камни” (например: разные endpoint’ы)
- рекомендуемые datasets (для демо)

Это решает: “в интернете всё есть, но никто не компоновал”.

---

## 62121–62170 — Mock Server OS (унифицированная имитация API)
Mock server поддерживает:
- маршруты (route) по шаблону OpenAPI
- статические ответы (fixtures)
- динамические ответы (генераторы)
- состояния (state machine): “создал заказ” → “оплатил” → “доставил”
- ошибки по расписанию (chaos testing): 429/500/timeout
- latency profiles

Главное: mock описывается декларативно (JSON/YAML), а не “ручными скриптами”.

---

## 62171–62220 — Fixtures & Datasets OS (данные как продукт)
Dataset содержит:
- список fixtures с типами и связями (entity graph)
- правила уникальности/дедупа
- версии (dataset v1/v2)
- “golden dataset” — эталон для демо и обучения

Примеры типов:
- Person, Company, Invoice, Payment, Ticket, Post, Product, Review, NewsEvent

Сценарии:
- “магазин” (товары→заказы→оплаты→возвраты)
- “офис” (задачи→письма→файлы→согласования)
- “портал сравнения” (оферы→правила→ранжирование)
- “новости” (источники→события→дедуп→итоги дня)

---

## 62221–62270 — Test Data Generators OS
Генераторы данных:
- deterministic режим (seed) → одинаковые данные при каждом запуске
- параметризация: язык, регион, валюты, форматы дат
- генерация связанных сущностей (order->payment->refund)
- генерация текстов для UI (заголовки, описания, отзывы)
- генерация “грязных данных” (ошибки, пропуски, дубликаты) для теста качества

---

## 62271–62320 — Record/Replay OS (проксирование и воспроизведение)
Когда есть доступ к реальному API, но нельзя каждый раз дергать прод:
- proxy пишет запрос/ответ (с редактированием PII)
- сохраняет как recording bundle
- replay воспроизводит ответы офлайн или через mock

Ключевые функции:
- redaction персональных данных
- нормализация id (stable mapping)
- повторяемость (same run → same outputs)
- подпись/хэширование записи

---

## 62321–62370 — Demo Tenants (общие демо‑аккаунты)
Для обучения и витрин:
- IFOS может держать “shared demo tenants” (ограничены квотами)
- авто‑reset: чистка данных по расписанию
- отдельные ключи/токены (низкие привилегии)
- анти‑абьюз: rate limits, captcha в UI, ограничение на exports

---

## 62371–62390 — Contract tests (согласование ожиданий)
Контрактные тесты:
- коннектор обязан проходить набор тестов против sandbox/mock
- проверяются поля, статусы, ошибки, ретраи
- результаты публикуются как “качество коннектора” (score)

Это создаёт вертикаль качества вместо “дикого поля”.

---

## 62391–62400 — Privacy & Compliance в песочнице
Правила:
- PII редактируется при record/replay
- sandbox datasets не содержат реальных данных
- экспорт recordings запрещён по умолчанию
- политики хранения: авто‑удаление по TTL
