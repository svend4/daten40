# IFOS · Certificate / PKI / mTLS (Управление сертификатами)

**Диапазон:** 75201–75600

## 1. Зачем нужен этот блок

Этот блок закрывает «криптографическую основу доверия» для IFOS:
- выпускает и обслуживает сертификаты (TLS) для сервисов, коннекторов, edge‑gateway, устройств и пользователей;
- поддерживает **mTLS** (двусторонний TLS) между компонентами (клиент и сервер взаимно проверяют сертификаты);
- обеспечивает **ротацию**, отзыв (revocation) и распространение доверенных цепочек (trust bundles);
- связывает сертификаты с **идентичностью** (тенант/организация/пространство/пользователь/устройство/сервис).

Идея: чтобы все «части операционной системы функций» могли безопасно разговаривать друг с другом без паролей и ручного обмена ключами.

## 2. Границы ответственности (что делает / не делает)

### 2.1 Делает
- Управляет **профилями сертификатов**: сроки, SAN, EKU, ключевые алгоритмы, политика авто‑продления.
- Организует **выпуск** (issuance) сертификатов:
  - через встроенную CA (минимально для dev/demo),
  - или через внешнюю CA / корпоративный PKI.
- Поддерживает жизненный цикл:
  - запрос (CSR) → выпуск → доставка → мониторинг → авто‑ротация → отзыв/замена.
- Публикует **trust bundles** (корневые и промежуточные цепочки) в нужные контуры:
  - облако, edge, offline‑кеш, локальные стеки.
- Логирует и аудирует события PKI (кто/когда/зачем выпустил/отозвал).

### 2.2 Не делает (намеренно)
- Не является полноценной «корпоративной CA‑платформой» уровня PKI‑suite (с сотнями UI‑форм).
- Не хранит секретные ключи в открытом виде: ключи должны жить в KMS/HSM/Secret‑Vault.
- Не заменяет IAM/SSO: сертификаты **связываются** с идентичностью из блоков Identity/Access.

## 3. Основные сущности (Data Model)

### 3.1 CertificateProfile (Профиль сертификата)
Определяет правила:
- назначение: server, client, device, connector, workload;
- алгоритм и параметры ключа (RSA/ECDSA), минимальные размеры;
- срок действия (validity_days);
- обязательные SAN/Subject шаблоны;
- EKU (serverAuth/clientAuth), разрешённые KU;
- авто‑продление (auto_renew_days);
- правила отзыва и компрометации.

### 3.2 Issuer (Эмитент)
Источники выпуска:
- Built‑in Dev CA (только для dev/demo);
- External CA (корпоративный PKI, облачные CA, ACME и др.).

У Issuer есть:
- endpoint/интеграционный тип,
- политики доступа,
- привязка к тенанту/организации,
- ссылка на хранилище ключей (KMS/HSM).

### 3.3 Certificate / CSR
- CSR хранится как артефакт заявки.
- Certificate хранит метаданные: отпечатки, срок, цепочку, назначение, владельца, статус (active/expiring/revoked).

### 3.4 TrustBundle
Набор доверенных корней/промежуточных:
- scope: tenant/org/workspace/global;
- стратегия распространения: cloud‑push, edge‑cache, offline‑sync;
- версия/ревизия для быстрого отката.

### 3.5 EnrollmentToken (Токен зачисления)
Короткоживущий токен для «ввода в доверие»:
- устройство/коннектор получает токен из админ‑консоли,
- по токену проходит первичную аутентификацию,
- затем получает свой первый сертификат.

## 4. Ключевые сценарии (User Journeys)

### 4.1 Самый простой (P0): ручной выпуск и доверенная цепочка
1) Админ создаёт Profile (serverAuth, 90 дней).
2) Создаёт Issuer (dev CA).
3) Сервис генерирует CSR и отправляет в IFOS.
4) IFOS возвращает сертификат и цепочку.
5) TrustBundle публикуется в нужные окружения.

### 4.2 Средний (P1): mTLS для коннекторов
1) Коннектор регистрируется и получает EnrollmentToken.
2) Коннектор запрашивает client‑cert (clientAuth) и получает его.
3) Edge‑Gateway/Runtime проверяет client‑cert, получает identity (tenant/org/workspace).
4) Политика доступа (IAM/Policy Engine) разрешает конкретные операции.

### 4.3 Продвинутый (P2): авто‑ротация + zero‑downtime
1) За N дней до истечения IFOS инициирует renewal.
2) Новый сертификат выдаётся параллельно (dual‑cert window).
3) Runtime применяет новый cert без простоя (hot reload).
4) Старый cert помечается как deprecated, затем revoked при необходимости.

## 5. API и интерфейсы (минимальный контракт)

> Формат ниже — «контракт‑набросок». Реализация может быть REST/gRPC.

- `POST /pki/profiles` — создать профиль
- `POST /pki/issuers` — зарегистрировать issuer
- `POST /pki/csrs` — загрузить CSR (с метаданными владельца)
- `POST /pki/certificates/issue` — выпуск (issuer + profile + csr)
- `POST /pki/certificates/renew` — продление
- `POST /pki/certificates/revoke` — отзыв
- `GET  /pki/trust-bundles` — получить trust bundle
- `POST /pki/trust-bundles/publish` — опубликовать bundle в контур
- `GET  /pki/audit` — аудит‑лог PKI событий

## 6. Интеграции с другими блоками IFOS

- **KMS / Secrets Rotation (74801–75200):**
  - хранение приватных ключей, ротация ключ‑материала, политики экспорта.
- **Identity Profiles / Enterprise IAM:**
  - маппинг `certificate_subject → principal_id`.
- **Edge Gateway / Offline Sync / Device Fleet:**
  - выпуск device‑cert, работа при потере связи, кеширование bundle.
- **Service Catalog / Topology:**
  - политика mTLS на уровне сервисов, владение, blast radius.

## 7. Риски и безопасность

- Компрометация ключа → быстрый отзыв, CRL/OCSP (минимально — список revoked отпечатков).
- Неправильные профили → запрет опасных EKU, SAN‑политики.
- DDoS на issuance → квоты и rate limit по org/workspace/issuer.
- Аудит → неизменяемый лог (append‑only).

## 8. План MVP (от простого к сложному)

### 8.1 P0 (минимум)
- Profiles + Issuer (dev) + CSR → certificate
- TrustBundle хранение/версии
- Простой аудит

### 8.2 P1 (рабочий продукт)
- EnrollmentToken
- mTLS для runtime/edge/коннекторов
- Авто‑renew по расписанию
- Интеграция с Policy Engine (кто может выпускать что)

### 8.3 P2 (enterprise)
- Поддержка внешних CA, ACME/SCEP‑подобные флоу
- OCSP/CRL интеграции
- HSM режим, разделение ролей (PKI admin vs app admin)
- Zero‑downtime ротация, массовые обновления

## 9. Acceptance Criteria (критерии готовности)
- Можно выпустить client‑cert и server‑cert по профилю и issuer.
- Можно включить mTLS между runtime и коннектором.
- Можно автоматически обновить cert до истечения срока.
- TrustBundle версионируется и доставляется в offline/edge контуры.
