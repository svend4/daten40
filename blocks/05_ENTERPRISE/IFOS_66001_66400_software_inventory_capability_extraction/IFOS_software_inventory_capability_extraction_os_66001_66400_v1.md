# IFOS — Software Inventory & Capability Extraction (GitHub/WP/Apps → IFOS Registry) (Блок 66001–66400)

Версия: v1 · Пакет: `IFOS_66001_66400_software_inventory_capability_extraction_os_pack.zip` · Дата: 2026-01-03

## 0) Зачем нужен этот блок

Ты описал проблему «интернет как дикое поле»: миллионы решений существуют, но:
- они плохо описаны и не сопоставимы;
- нет понятной “вертикали” функций (что умеет, чем лучше, где применить);
- нет стабильного реестра и матрицы совместимости.

**Software Inventory & Capability Extraction** — это слой IFOS, который умеет:
1) собирать “сырые” источники (репозитории, плагины, приложения),
2) извлекать метаданные и “функции/способности”,
3) нормализовать в единый словарь IFOS (capabilities),
4) строить матрицы сравнения, рейтинги, и “кластеры”.

Это фундамент для витрин, сравнения, one-click пакетов и обучения.

## 1) Что делает (функции)

1) **Inventory sources**: GitHub repos, WordPress plugins (каталоги), “app listings” (описания).
2) **Metadata extraction**:
   - название/версия/лицензия
   - зависимости (composer/package.json/requirements)
   - ключевые слова/категории
   - поддерживаемые API/протоколы (OAuth, Webhooks, REST)
3) **Capability inference**: выводит IFOS-capabilities из описаний и структуры проекта.
4) **SBOM-light**: список пакетов/компонентов (минимальный).
5) **Compatibility matrix**: совместимость с коннекторами/раннером/профилями окружения.
6) **Dedup & identity**: распознаёт дубль/форк/переименованный проект.
7) **Quality signals**: активность, релизы, тесты, популярность, документация.
8) **Registry sync**: публикует результат в IFOS Registry (как карточки/пакеты).

## 2) Что НЕ делает

- Не гарантирует правильность inference на 100%: нужны ручные правки и community feedback.
- Не скачивает закрытый код из маркетплейсов.
- Не является полноценной SCA/сканер-платформой, но задаёт структуру и минимальные поля.

## 3) Данные и сущности (минимальная схема)

- `SoftwareAsset`: репозиторий/плагин/листинг
- `Capability`: нормализованная способность IFOS (например `cap:webhook_receiver`)
- `ConnectorProfile`: что нужно для подключения (OAuth scopes, keys)
- `Compatibility`: (asset × profile) → статус + причины
- `QualitySignals`: метрики доверия
- `Provenance`: откуда взято и когда

## 4) Пайплайн (простое → среднее → сложное)

### 4.1 Простое (P1)
- импорт списка URL/ID
- парсинг метаданных и ключевых слов
- ручное назначение capabilities

### 4.2 Среднее (P2)
- полуавтоматический inference capabilities
- дедуп/форки
- базовая матрица совместимости

### 4.3 Сложное (P3)
- авто‑кластеризация по “функциональным словам”
- активное обучение по фидбеку пользователей
- автоматическая генерация витрин/сравнений

## 5) Матрица совместимости (пример)

Статусы:
- `OK` — подходит в профиле `demo/dev/edge`
- `PARTIAL` — нужно донастроить/нет одного коннектора
- `BLOCKED` — несовместимо (лицензия/секреты/архитектура)

Причины:
- `missing_connector`
- `requires_private_keys`
- `license_restriction`
- `needs_webhooks_public_url`
- `needs_db_feature`

## 6) Дыры/что ещё не закрыто

- Нужна политика источников и кэширование (чтобы не DDoS каталоги).
- Нужны правила лицензий (что можно индексировать/копировать).
- Нужна “семантика функций” (онтология) для качества inference.
- Нужна репутация источников и подтверждение экспертами.
